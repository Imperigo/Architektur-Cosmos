# Architecture Office 2045 Strategy Guideline

Source context: private strategic PDF `Architekturbuero_2045_AI_Strategiebericht.pdf`,
provided as project context on 2026-05-21.

This document stores what Architecture Cosmos should learn from the broader
office strategy. Architecture Cosmos remains its own project: a public-safe
architecture atlas, archive, database foundation and future Blender/ArchiCAD
reference layer. The office strategy is used here only as a guiding compass.

## Core Lesson

AI makes raw output cheaper: variants, renderings, summaries, first reports,
basic plan extraction and repetitive coordination will become easier to produce.
The durable value moves toward responsibility, orchestration, verification,
data governance, source quality, cultural judgment and buildable knowledge.

For Architecture Cosmos this means:

- do not become only a visual atlas or image demo;
- become a verified reference and decision infrastructure;
- treat every entry as structured knowledge, not just content;
- keep human review, source trails and rights gates central;
- make 3D/model data useful for downstream design tools, not only impressive on
  the website.

## Target Model For Architecture Cosmos

Architecture Cosmos should support the strategy model:

1. **Orchestrator**
   The atlas and Brain coordinate sources, entries, media, analysis, 3D layers,
   rights status, review tasks and future publication decisions.

2. **Platform**
   The database becomes a reusable product layer: entries, taxonomies, API
   access, Blender/ArchiCAD references, model packages and analysis layers.

3. **Haltung / Cultural Position**
   The project must preserve architectural judgment: material, atmosphere,
   tectonics, context, history and authorship. It should not read as generic
   automation.

## Agent Stack Translation

The broader office strategy lists agent roles such as parcel scout, briefing
memory, BIM operator, cost/carbon brain, visual engine, site brain and QA gate.
For Architecture Cosmos, translate them as follows:

- **Archive Scout:** finds candidate projects, public-safe sources and missing
  references.
- **Source Trail Agent:** checks source reliability, rights status and citation
  completeness.
- **Entry Curator:** turns raw research into architecture-specific briefs,
  taxonomies, relations and study prompts.
- **3D / Model Planner:** defines massing, structure, tectonic, facade, site and
  material layers for Blender/ArchiCAD use.
- **Analysis Agent:** extracts material, structure, roof form, circulation,
  typology, urban/landscape context and tectonic tags.
- **Media Rights Gate:** decides public display, private-only use, link-only
  references or rejection.
- **Brain QA Gatekeeper:** the most important agent; verifies, audits, blocks
  unsafe output and asks for approval before commits, cloud writes, publication
  or asset uploads.

## Data Model Implications

Every mature Architecture Cosmos entry should move toward:

- verified metadata and source trails;
- public/private media distinction;
- source reliability and rights status per source/asset;
- material, structure, tectonic, typology, context and program tags;
- relation graph to other entries;
- model packages for web preview and Blender/ArchiCAD import;
- confidence levels and review status for AI-generated analysis;
- clear separation between public atlas data and private research material.

## Product Principles

1. **No Tool Fetish**
   Build only tools that improve archive quality, learning value, design-tool
   usefulness, risk control or review speed.

2. **QA Before Autonomy**
   Agents may research, draft and diagnose. They do not silently publish, upload
   private material, rewrite source data or claim certainty.

3. **Source-Based Intelligence**
   Every public claim should be traceable. AI summaries without source trail are
   draft notes, not database truth.

4. **Rights-Aware By Default**
   Public display requires `own_work`, `licensed` or `public_domain`.
   Copyright-sensitive sources may inform private analysis only when the rights
   gate marks them accordingly.

5. **Design Tool Readiness**
   3D output should be layer-aware from the start: site, mass, structure,
   envelope, materials, circulation, tectonics and analysis JSON.

6. **Human Architectural Judgment**
   The system should learn from data, but final quality depends on reading
   atmosphere, scale, material, construction, cultural context and responsibility.

## Roadmap Adaptation

Near-term Architecture Cosmos work should prioritize:

1. strengthen the Brain as a QA and review system;
2. improve source and rights coverage for existing entries;
3. make the database UI clearly separate public review from dev/private tools;
4. build robust entry enrichment and media-audit workflows;
5. deepen the 3D/model layer standard for Blender and ArchiCAD;
6. keep mobile/responsive UI stable before adding heavy effects;
7. prepare private library/upload architecture only behind real auth and
   quarantine.

## Risk Register

- **Hallucination:** require source trail, confidence and review status.
- **Rights leakage:** keep private/copyright assets out of public rendering and
  R2 public paths.
- **Data chaos:** keep schemas, tags, naming and model package paths consistent.
- **Cold software feel:** preserve architectural language, material readings and
  project-specific visual identity.
- **Automation without responsibility:** Brain suggests and audits; owner approves
  major changes.
- **Performance drift:** immersive UI is valuable only while smooth, readable and
  accessible.

## Operating Rule

When future Architecture Cosmos features conflict with this guideline, prefer:

1. verified over flashy;
2. rights-safe over complete;
3. structured over improvised;
4. reviewable over autonomous;
5. architecture-specific over generic AI output.
